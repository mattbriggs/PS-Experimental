About the Heretto AI Solutions Pattern LibraryThe Heretto AI Solutions Pattern Library provides reusable consulting patterns for designing, scoping, implementing, and governing AI solutions that use Heretto as a trusted content and knowledge source.PurposeThis library helps customer-facing teams move from vague AI interest to concrete solution architectures. It does so by describing common problems, contexts, forces, solutions, risks, reference diagrams, implementation guidance, and success measures.The library supports discovery calls, solution design, statement-of-work scoping, proof-of-concept planning, customer education, support escalation, and product feedback.What this library isA consulting enablement library for Heretto AI solutions, organized as a pattern language in the tradition of Christopher Alexander and the Gang of Four. Each pattern names a recurring problem, the forces that make it hard, a repeatable solution, related patterns, and the resulting context.What this library is notĄIt is not a generic AI architecture catalog. Every pattern assumes Heretto-managed DITA content as the governed substrate.ĄIt is not product documentation for Heretto features. See product docs for feature-level instructions.ĄIt is not a procurement document. It is a working set of consulting assets.Core thesisHeretto supports AI solutions not simply by serving content to models, but by governing the entire lifecycle of knowledge: source discovery, content creation, DITA transformation, metadata and taxonomy governance, publication, retrieval, agent access, workflow control, evaluation, and continuous improvement.How the library is organizedThe library has four functional layers, mapped to nine pattern groups:1.Source to Heretto.  Discovery, creation, and transformation patterns.2.Heretto governance.  Metadata, lifecycle, taxonomy, and discovery patterns.3.Heretto to AI.  Agent-ready content, integration, and workflow safety patterns.4.Feedback to Heretto.  Evaluation, gap detection, and optimization patterns.Related informationAudience and uses  (on page 9)How to use this library  (on page 13)Reference architecture  (on page 18)Chapter 2. How to use this libraryUse this procedure to move a customer engagement from initial interest to a scoped, supportable AI solution backed by Heretto-governed content.You should know the customer's general business, the Heretto products in use, and the customer's stated AI interest. You should have access to a Heretto tenant for shared examples.The library is designed to be read non-linearly. Start at the lifecycle process that matches the engagement phase, then drill into pattern groups as questions arise.1.Identify the engagement phase.Phases are: discover, assess, design, implement and pilot, govern and improve. See the Lifecycle overview process.2.Open the matching lifecycle process topic.Each process lists inputs, decisions, outputs, and the patterns relevant at that phase.3.Read the relevant pattern group overview.Each group overview describes the problem space, the dependencies among patterns in the group, and the patterns to consider first.4.Read individual patterns.Every pattern follows the same structure: intent, context, problem, forces, solution, structure with diagram, resulting context, discussion, and related patterns.5.Use the patterns to draft customer deliverables.Use the playbooks for repeatable deliverables. Cite the patterns in customer documents to anchor terminology and decisions.6.Record gaps and feedback.If a pattern does not fit the customer situation, add a comment in the engagement notes and submit a change request to the library owner. The library improves through use.You leave the engagement with a named, scoped, and defensible solution architecture rather than a free-form AI promise.After the engagement, file the success measures, what worked, and what did not. The library team uses this feedback to update patterns and add new ones.Chapter 5. DITA and information-typing conventionsThis library uses DITA 1.3 base topic types only. Pattern, principle, and process information types are expressed through the @outputclass  attribute, with no custom specialization.Why no specializationSpecialization adds long-term maintenance burden and depends on tooling support. By using base DITA 1.3 with @outputclass, the library remains portable to any conformant DITA processor. Stylesheets can render the variants distinctively without DTD changes.Topic-type mappingInformation type Base topic @outputclass Pattern concept pattern Principle concept principle Process task process Group overview concept group-overview Playbook task playbook Concept concept (none) Task task (none) Reference reference (none) Troubleshooting troubleshooting (none) Glossary entry glossentry (none) Horn block mappingRobert E. Horn information mapping in this libraryRobert E. Horn's Information Mapping method shapes how each topic is chunked, labeled, and ordered. The method's discipline helps readers locate information by purpose rather than by paragraph order.What Information Mapping contributesInformation Mapping defines small set of information types and rules for organizing them: chunking, relevance, labeling, consistency, integrated graphics, accessible detail, and hierarchy. This library applies those rules within DITA topics so each block has one purpose and one label.Block-level rules applied hereĄChunking.  Each <section>  holds one Horn block type and one idea.ĄRelevance.  A block contains only information consistent with its block type.ĄLabeling.  Every block has a short, descriptive title.ĄConsistency.  Similar information appears in the same block type and order across topics.ĄIntegrated graphics.  Diagrams sit beside the explanation they support, not at the end.ĄAccessible detail.  Each block can be read independently.ĄHierarchy.  Topics group into maps; sections group into topics.Block types used in patternsA pattern topic uses the following Horn blocks, expressed as DITA sections:1.Intro.  The pattern's intent and shortest summary.2.Context.  Where the pattern applies. A fact  block.3.Problem.  The recurring tension. A fact  block.4.Forces.  Competing concerns. A classification  block.5.Solution.  The repeatable response. A principle  block.6.Structure.  The component view with diagram. A structure  block.7.Resulting context.  What becomes possible. A fact  block.8.Discussion.  Caveats and notes. A fact  block.Block types used in processesA process topic (DITA task with @outputclass="process") uses these Horn blocks:1.Intro.  The purpose of the process.2.Inputs.  What you must have. A fact  block.3.Decisions.  The choices the process makes. A classification  block.4.Procedure.  The numbered actions. A procedure  block (DITA <steps>).5.Outputs.  What you produce. A fact  block.6.Related patterns.  Patterns used by this process.Why Horn plus DITADITA enforces topic-type discipline at the document level. Horn enforces information-type discipline at the block level. Together they make content predictable for human readers, search systems, and AI retrievers.Related informationDITA and information-typing conventions  (on page 14)Chapter 7. Reference architectureThe reference architecture has four layers: source to Heretto, Heretto governance, Heretto to AI, and feedback to Heretto. The pattern library is organized around these layers.OverviewSource material flows into Heretto through controlled discovery, creation, and transformation. Heretto governs the content with metadata, lifecycle, taxonomy, and discovery. Agents consume Heretto content through retrieval, MCP, and graph-augmented patterns. Failures and signals from agents and search return to Heretto as content, metadata, and taxonomy work.Architecture diagramThe four layersTable  3. Architecture layersLayer Responsibility Source to Heretto Discover, qualify, and transform source content into governed DITA 1.3 topics with provenance. Heretto governance Manage metadata, lifecycle, subject schemes, taxonomy, discovery, and validation. Heretto to AI Expose approved content to retrieval, MCP, and graph-augmented agents within enforced boundaries. Feedback to Heretto Convert AI and search signals into content, metadata, taxonomy, and governance work. Why the loop mattersAn AI system is only as good as the content pipeline feeding it. The loop ensures that content quality improves with use rather than degrading over time. Without the loop, the system industrializes its own errors.Related informationSource to Heretto  (on page 19)Heretto governance layer  (on page 21)Heretto to AI  (on page 22)Feedback loop  (on page 23)Source to HerettoThe source-to-Heretto layer covers how content enters the CCMS in a controlled way, with provenance, classification, and conversion.What this layer doesSource material exists in many forms: Word, PDF, Markdown, HTML, SharePoint, Confluence, GitHub, support tickets, transcripts, legacy XML, and tribal knowledge. Governing principlesSix principles govern the pattern library. They guide pattern selection, trade-off decisions, and conflict resolution when patterns appear to contradict one another.Why principlesPatterns describe what to do in recurring situations. Principles describe why. When a customer situation does not match any single pattern, principles guide judgment.The six principles1.Content is the substrate, not the surface.  (on page 25)2.Provenance precedes publication.  (on page 26)3.Metadata is the API to your content.  (on page 26)4.Taxonomy is vocabulary governance.  (on page 27)5.The agent is bounded by the gate.  (on page 28)6.Failure is feedback.  (on page 28)Content is the substrate, not the surfaceAI features are surfaces. Governed content is the substrate that determines whether those surfaces produce useful, defensible answers.Principle statementTreat AI features as consumers of governed content, not as replacements for content work. Invest in content structure, metadata, and lifecycle before investing in agents.ImplicationsĄScope AI engagements around content readiness, not model selection.ĄTreat retrieval quality problems as content problems first and model problems second.ĄPlan content investment as a precondition for AI scale.If you ignore this principleWhen the substrate is weak, the surface becomes a faster way to deliver bad answers.Related informationProvenance Precedes Publication  (on page 26)Metadata Is The Api  (on page 26)Provenance precedes publicationEvery content object should carry its source, transformation history, and approval state before it becomes available for publication or AI use.Principle statementAttach a provenance envelope to content the moment it enters Heretto. Block publication and AI exposure until the envelope is complete.ImplicationsĄRequire source ID, source system, creation mode, and review status on import.ĄMark AI-generated drafts distinctly until they pass review.ĄPreserve provenance through transformations and through publication.If you ignore this principleWithout provenance, no one can defend an AI answer, retire a deprecated source, or trace a customer-facing claim back to its evidence.Related informationContent Is Substrate  (on page 25)Failure Is Feedback  (on page 28)Metadata is the API to your contentAgents, search systems, dashboards, and personalization all consume content through metadata. Treat metadata as a contract, not as an afterthought.Principle statementDefine a tiered metadata model with required fields for identity, governance, product, audience, taxonomy, publication, and AI use. Enforce completion before exposure.ImplicationsĄRequired metadata is a publication and AI gate, not a preference.ĄAuthor tools must make required fields obvious and assistive.ĄMetadata schema changes are governance events, not silent migrations.If you ignore this principleWithout metadata, retrieval becomes guessing and personalization becomes impossible.Related informationTaxonomy Is Vocabulary Governance  (on page 27)Agent Bounded By Gate  (on page 28)Taxonomy is vocabulary governanceTaxonomy is not decoration. It is the controlled vocabulary that lets search, retrieval, personalization, and AI generalize across topics.Principle statementManage taxonomy as governed vocabulary with preferred labels, alternate labels, and relationships. Use SKOS to share the vocabulary across Heretto, AI systems, and analytics.ImplicationsĄCapture user language as alternate labels rather than tolerating drift.ĄRound-trip taxonomy between Heretto and authoritative tools.ĄMeasure taxonomy use and prune unused or duplicated concepts.If you ignore this principleWithout governance, taxonomy splinters into a thousand local dialects.Related informationMetadata Is The Api  (on page 26)Failure Is Feedback  (on page 28)The agent is bounded by the gateAgents may draft, retrieve, summarize, classify, and propose actions. Agents must not publish, send customer messages, or modify authoritative content without a human gate.Principle statementDefine explicit tool boundaries for every agent and require a Human Review Gate for any action with external consequences.ImplicationsĄDefault agent permissions are read-only.ĄWrite actions require a named approver and a logged decision.ĄTool boundaries are version-controlled artifacts, not runtime guesses.If you ignore this principleAn unbounded agent can amplify errors faster than humans can detect them.Related informationProvenance Precedes Publication  (on page 26)Metadata Is The Api  (on page 26)Failure is feedbackFailed retrievals, search misses, low-confidence answers, and unresolved tickets are signals. Convert them into named work items, not lessons forgotten.Principle statementRun an evaluation bench and a content performance dashboard. TroubleshootingThis appendix covers common failure modes across the Heretto AI pipeline. Each topic uses the DITA 1.3 troubleshooting type with condition, cause, and remedy.Troubleshooting topicsĄHallucinated citations  (on page 155)ĄMissing metadata blocks retrieval  (on page 156)ĄTaxonomy drift  (on page 156)ĄAgent over-reach  (on page 157)ĄLocalization parity failure  (on page 158)Hallucinated citationsAn agent answer cites a topic that does not exist, a fragment that does not match, or a deprecated topic.ConditionUsers report citations that 404, point to deprecated content, or describe content that the cited topic does not contain.CauseThe agent generated the citation rather than rendering a retrieval result, or retrieval returned a chunk without a valid anchor.Resolve hallucinated citations1.Validate every citation against the canonical URL store before rendering.2.Drop citations that fail validation rather than rendering broken links.3.Apply the Citation Anchor Strategy pattern to ensure stable anchors.4.Apply the Retrieval with Provenance pattern so the agent renders citations from retrieval, not from generation.5.Add a regression case to the evaluation bench so the failure is detected next time.Related informationCitation Anchor Strategy  (on page 91)Retrieval With Provenance  (on page 101)Agent Evaluation Bench  (on page 122)Missing metadata blocks retrievalTopics that should be available to the agent are not returned by retrieval, even when the content clearly matches the query.ConditionSearch and agent retrieval miss topics that match the query and should be in scope.CauseThe topics are missing required metadata fields, so the retrieval filter or lifecycle filter excludes them. Or, the topics carry incorrect metadata values.Resolve missing-metadata exclusion1.Apply the Metadata Governance Model pattern to confirm required fields per use case.2.Apply the Metadata Completion Gate pattern to block missing topics from publication.3.Run a metadata completeness report from the Heretto Discovery Surface pattern.4.Backfill metadata for the affected topics.5.Add a regression case to the evaluation bench.Related informationMetadata Governance Model  (on page 69)Metadata Completion Gate  (on page 70)Heretto Discovery Surface  (on page 75)Taxonomy driftUsers describe problems in language that no longer matches the taxonomy, and retrieval suffers.ConditionSearch and AI miss content because user vocabulary diverges from the taxonomy. Authors stop tagging because the tags do not fit user language.CauseThe taxonomy aged without a refresh process. New product names, customer-coined terms, and competitor language entered the system, but the taxonomy did not.Resolve taxonomy drift1.Run the Taxonomy Drift Detector pattern to surface high-volume unmatched terms.2.Apply the Synonym and Variant Vocabulary pattern to add alternate labels.3.Apply the Continuous Taxonomy Improvement pattern to add a monthly review cadence.4.Update the SKOS scheme and round-trip back to Heretto.5.Notify authors and agents of the new labels.Related informationTaxonomy Drift Detector  (on page 85)Synonym And Variant Vocabulary  (on page 83)Continuous Taxonomy Improvement  (on page 125)Agent over-reachAn agent takes actions or accesses content beyond what its tool boundary should allow.ConditionAudit logs show tool calls outside the agent's documented permissions. A customer-facing message goes out without a review record.CauseTool boundaries are missing, outdated, or unenforced. The review gate is bypassed or absent.Resolve agent over-reach1.Stop the affected agent and identify the bypassed control.2.Apply the Agent Tool Boundary pattern to declare permissions explicitly.3.Apply the Human Review Gate pattern for any action with external consequences.4.Apply the Observability and Audit Trail pattern to detect future over-reach.5.Re-enable the agent only after the boundary and gate are verified.Related informationAgent Tool Boundary  (on page 104)Human Review Gate  (on page 110)Observability And Audit Trail  (on page 117)Localization parity failureLocalized customers see different content from source customers, or AI answers differ by locale for the same question.ConditionSource topics have been updated but localized topics have not. Customers in one locale receive answers based on stale or missing content.CauseThe localization pipeline did not pick up source changes, or the agent retrieved source-only content for a locale-tagged query.Resolve localization parity failure1.Apply the Localization Parity Agent pattern to compare source and localized topics.2.Route flagged topics to the localization queue with priority.3.Confirm retrieval is locale-aware and excludes stale localized content where required.4.Apply the Lifecycle State Filter pattern to gate stale localized content.5.Backfill localizations and rerun the parity agent.Related informationLocalization Parity Agent  (on page 127)Lifecycle State Filter  (on page 72)Validation And Compliance Harness  (on page 114)Chapter 22. AgentAn autonomous or semi-autonomous software actor that uses tools, retrieves content, and produces outputs to accomplish user goals.Chapter 23. Agent-ready topicA DITA topic packaged with metadata, summary, retrieval chunks, citation anchors, and related concepts so that agents can use it predictably.Chapter 24. A2AAgent-to-agent. An interaction model where specialized agents hand off tasks to each other, often using shared artifacts.Chapter 25. Authoring contractA small, enforceable set of obligations for human authors writing Heretto-ready DITA content.Chapter 26. CCMSComponent content management system. The category of system in which DITA topics and maps are authored, managed, and published.Chapter 27. ChunkA retrieval unit produced from a DITA topic. Chunks should respect topic structure rather than arbitrary token counts.Chapter 28. Citation anchorA stable identifier and URL fragment that allows agent citations to land at the right section of a topic.Chapter 29. ConrefDITA's content reference mechanism. A topic can reference content from another topic by ID.Chapter 30. DITADarwin Information Typing Architecture. An OASIS standard for modular topic-based technical content.Chapter 31. DITA mapA DITA document that organizes topics into publications using references.Chapter 32. Evaluation benchA versioned suite of cases that tests retrieval, citation, faithfulness, and workflow quality on every change.Chapter 33. FaithfulnessThe property of an AI answer being supported by the retrieved evidence. Faithfulness is an evaluation metric, not a guess.Chapter 34. GovernanceThe set of processes, roles, and rules that determine how content is created, approved, exposed, retired, and improved.Chapter 35. GraphRAGGraph-augmented retrieval augmented generation. A retrieval approach that uses graph relationships also to vector similarity.Chapter 36. HerettoA component content management system that supports DITA 1.3 authoring, metadata, lifecycle, and publication.Chapter 37. HITLHuman-in-the-loop. A control where a human reviews and approves an agent action before it has external consequences.Chapter 38. Information mappingRobert E. Horn's method for chunking, labeling, and ordering information into named block types.Chapter 39. KeyrefDITA's key reference mechanism. A way to indirect references through a key so that targets can vary by publication.Chapter 40. Lifecycle stateThe state of a topic in its publication lifecycle. States include draft, in review, approved, deprecated, and archived.Chapter 41. MCPModel Context Protocol. A standard interface for exposing resources, tools, and prompts to AI agents.Chapter 42. MetadataStructured information attached to a topic or element. Metadata supports filtering, retrieval, personalization, and governance.Chapter 43. PatternA named, reusable response to a recurring problem in a defined context, with stated forces, solution, and resulting context.Chapter 44. PlaybookA bundle of patterns and processes packaged as a repeatable consulting engagement.Chapter 45. PoolPartyA commercial semantic platform for managing SKOS taxonomies and ontologies.Chapter 46. PrincipleA statement that guides decisions when patterns do not apply directly. Principles capture why patterns work.Chapter 47. ProcessA repeatable sequence of decisions and actions. In this library, a process is a DITA task with outputclass set to process.Chapter 48. ProtŽgŽAn open-source ontology editor that supports OWL, RDF, and SKOS.Chapter 49. ProvenanceInformation that records where content came from, how it was transformed, and who approved it.Chapter 50. RAGRetrieval-augmented generation. A pattern where an AI model retrieves source passages and grounds its answers in them.Chapter 51. RetrievalThe process of finding the most relevant passages from a corpus to answer a query. Retrieval can be vector-based, keyword-based, graph-based, or hybrid.Chapter 52. ScorecardA reusable assessment instrument. In this library, the AI Readiness Scorecard scores content, metadata, taxonomy, governance, and integration.Chapter 53. SKOSSimple Knowledge Organization System. A W3C standard for representing controlled vocabularies and concept schemes.Chapter 54. Subject schemeA DITA mechanism for defining controlled values for metadata and classification.Chapter 55. TopicA self-contained unit of content with a title and a single purpose. The basic building block of DITA.Chapter 56. Topic typeA DITA classification of a topic by its purpose. This library uses concept, task, reference, troubleshooting, and glossentry.Chapter 57. Tool boundaryThe explicit set of tools, schemas, and permissions that an agent is allowed to use.
